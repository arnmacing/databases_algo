#### Notes
```text
По этим цифрам видно, что качество построения первого уровня нестабильно и именно это потом раздувает суммарный размер вторичных таблиц. 
Суммарный размер второго уровня это сумма квадратов размеров бакетов `Σ n_j²`, а число коллизий первого уровня — сумма `(size(bucket) - 1)` по непустым бакетам.
Обе метрики собираются прямо в бенчмарке, а сама структура строится так, что сначала подбирается первая хеш-функция, затем для каждого бакета создаётся вторичная таблица размера `n_j²`. 
Поэтому аномалия `n = 200000`, где `secondaryTableSize = 2_136_632_480` это расширение примерно в **10683** раз относительно числа ключей, а число коллизий `427_316_240` вообще физически не похоже на корректную метрику для одного построения на 200 тысяч ключей. 
Для сравнения, при 400k, 600k и 800k значения уже выглядят на порядок разумнее, хотя и там качество распределения всё ещё скачет. 
Это наводит на две мысли. Первая: либо в дополнительных метриках есть ошибка агрегации или дублирование счётчиков на нескольких вызовах, потому что для одного построения коллизии первого уровня не могут быть настолько больше `n`; 
Вторая: даже если часть чисел реальна, всё равно слабое место именно в выборе первой хеш-функции и в цене построения вторичных таблиц. 
В коде сейчас критерий принятия первой функции очень мягкий: цикл останавливается, как только `Σ n_j² <= 4n`, после чего сразу строятся все вторичные таблицы . 
Самая полезная оптимизация здесь **искать не первую подходящую первую хеш-функцию, а лучшую из нескольких кандидатов**, например из 8 или 16 случайных вариантов выбирать ту, у которой минимальна `Σ n_j²` или хотя бы минимален максимальный бакет. Это даст выигрыш и по памяти, и по времени построения, потому что вторые уровни станут меньше и проще. 
Вторая идея **не создавать вторичную таблицу размера `n_j²` для бакетов размера 1 и 2 по общему правилу**: для одного ключа можно хранить его напрямую без отдельной большой схемы, для двух ключей можно использовать маленький специализированный контейнер. 
Также можно **убрать лишние промежуточные массивы** `keysByBucket`, `valuesByBucket`, `pos`, а потом для каждого бакета ещё раз массивы во втором уровне .
```
